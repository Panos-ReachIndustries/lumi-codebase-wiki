# V2 arbiter

`protocol_arbiter_v2` is the production arbiter. It mirrors v1's external behaviour — same Kafka topics, same command vocabulary, same monitor-spawning hooks — but replaces the nested `stage/step/action` tree with a **flat list of numeric instruction IDs**. The relay points its `Arbiter` monitor type at `arbiter_v2.py` (`Lumi-AI-Continuous/monitor_relay/monitor.go:113`), so this is what runs whenever an experiment starts.

## Flat-instruction model (vs v1)

In [v1](v1.md), a protocol was a nested tree and progression came from position in that tree. V2 collapses that to:

```json
{
  "version": 2,
  "instructions": [
    { "id": 10, "label": "Weigh salicylic acid", "metadata": {} },
    { "id": 20, "label": "Add ethanol",          "metadata": {} },
    { "id": 35, "label": "Heat to 70°C",         "metadata": {} }
  ]
}
```

Order is determined solely by sorting `id` ascending. Sparse IDs (10, 20, 35) are deliberate — they leave room for inserting steps without renumbering. `InstructionIndex.get_next_id` in `Lumi-AI-Continuous/protocol_arbiter_v2/instructions.py:44-66` is the single source of truth for "what's next?". Loaders in `instructions.py:95-110` accept both the new format and the legacy nested JSON via `load_v2_protocol_from_legacy`, so v1 protocols still play.

`ProtocolStateV2` in `protocol_arbiter_v2/state.py:40-57` carries `current_instruction_id`, `completed_instruction_ids: Set[int]`, and per-instruction `InstructionRuntime` records (state, started_at, completed_at, paused periods).

## Entry point

```bash
python protocol_arbiter_v2/arbiter_v2.py --config <path>
```

`Lumi-AI-Continuous/protocol_arbiter_v2/arbiter_v2.py:24-27` emits a `phase_marker("started")` via [Common.profiling](../common.md) before any heavy imports, so the relay can time boot phases. The `ProtocolArbiterV2` class wires up the Kafka consumer/producer, fetches the protocol from S3 (or `LOCAL_STORE_DIR` for `run_dev.py`), builds the instruction index, and starts the run loop.

## Topics consumed

All topic names are env-overridable via `protocol_arbiter_v2/kafka_io_v2.py:16-20`:

- `PROTOCOL_ARBITER_COMMANDS_TOPIC` (default `protocol-arbiter-commands`) — user commands keyed by experiment ID. `arbiter_v2.py:636-656` does manual `assign()` + `seek()` to the experiment's `initialTimestamp` so a restarted arbiter replays only its own commands.
- `AGENT_PROTOCOL_SYNC_TOPIC` (default `agent-protocol-sync`) — shared back-channel for inter-arbiter coordination. Subscribed at `arbiter_v2.py:680`. Filtered by Kafka key on receipt to avoid CPU pathologies (every arbiter would otherwise decode every message — see `Testing/test_arbiter_cpu_pathologies.py`).
- Monitor data topics — added on demand via `addListener` commands, consumed by `CuratorV2`.

## Topics published

`kafka_io_v2.py` is the single place that knows topic names and `produce()`-then-`flush()` patterns:

- `PROTOCOL_ARBITER_STATUS_TOPIC` (default `protocol-arbiter-status`) — lifecycle status (`publish_status`).
- `PROTOCOL_ARBITER_HISTORY_TOPIC` (default `protocol-arbiter-history`) — ledger entries (`publish_history`).
- `PROTOCOL_ARBITER_RESPONSES_TOPIC` (set per-environment) — structured responses to the web app's VLC. Required `x-type: lumi_live_message` header; messages without it are silently ignored downstream (`kafka_io_v2.py:48-71`).
- `monitor-commands-<id>` and `monitor-events-<arbiterId>` — same per-experiment fan-out as v1, used to spawn and address individual monitors.

## Predicate evaluation

V2 still treats the protocol JSON as the source of truth for what "complete" means — completion is recognised when `HistorianV2` (`historian_v2.py`) sees an event in `history_cache["protocol_actions"]` whose `instructionId` matches the current instruction. `state.complete_current_instruction` then flips the runtime state and the index advances. There is no first-class predicate ADT in v2; that's what the [v3](v3.md) redesign brings. Block-based completion logic lives in the custom-agent JSON consumed by per-monitor processes, not in the arbiter itself.

## Why it's the production default

- **Standalone.** No imports from `protocol_arbiter` — owns its own curator, historian, history, kafka helpers, and lab manager.
- **Same wire format as v1.** Drop-in replacement; Kafka topics, command vocabulary, and ledger shape unchanged.
- **Cleaner state.** Sparse IDs eliminate the renumber-the-world problem. Instructions can be reordered, paused, resumed, and re-evaluated without touching a tree.
- **Tested in anger.** `protocol_arbiter_v2/Testing/` carries 14 pytest files covering instructions, state transitions, capability resolution, capacity calculation, manifest publishing, and the CPU pathology guard.

## Tests

`protocol_arbiter_v2/Testing/` (run via `pytest protocol_arbiter_v2/Testing -q`):

- `test_state_and_instructions_v2.py` — index building, `get_next_id`, runtime state mutations.
- `test_historian_v2.py` — instruction-event mapping into ledger entries.
- `test_arbiter_cpu_pathologies.py` — guards against the shared-topic decode storm.
- `test_capability_resolver*.py`, `test_capacity_calculator*.py` — fleet-shaping invariants.
- `test_config_and_kafka_v2.py` — env resolution, consumer/producer factories.
- `test_ui_gateway.py`, `test_manifest_publish.py` — the VLC response surface.
