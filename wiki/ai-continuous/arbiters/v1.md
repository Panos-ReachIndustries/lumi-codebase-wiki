# V1 arbiter

The v1 protocol arbiter is the original orchestrator for Lumi experiments. It spawns monitor processes, listens to their results on Kafka, decides when each protocol step has completed, and writes a replayable ledger to S3. It is the direct ancestor of [v2](v2.md) and [v3](v3.md), and remains in the tree for backwards compatibility — but [v2](v2.md) is what production runs.

## What it is

A long-lived Python process, one per running experiment. The class `ProtocolArbiter` in `Lumi-AI-Continuous/protocol_arbiter/arbiter.py:54` owns three Kafka clients, a thread for each subsystem, and the in-memory `protocol` dict that mirrors the JSON pulled from S3.

Three roles, three files:

| File | Class | Job |
|------|-------|-----|
| `protocol_arbiter/arbiter.py` | `ProtocolArbiter` | Main controller — Kafka, lifecycle, command dispatch, monitor management. |
| `protocol_arbiter/curator.py` | `DataCurator` | Subscribes to monitor output topics, normalises payloads into `history_cache`. |
| `protocol_arbiter/historian.py` | `ExperimentHistorian` | Walks `history_cache`, marks stage/step/actions complete, uploads ledger + protocol snapshots to S3. |

## The stage / step / action model

A v1 protocol is a nested tree. `arbiter.py:165-173` initialises every level with `completed = False`:

```
protocol
└── stages[]
    └── steps[]
        └── actions[]
```

Completion bubbles up: an action gets a `startTime` / `endTime` / `completed=True` written by the historian when its triggering event arrives in the cache (`historian.py:140-142`). When all actions in a step are done, the step closes; when all steps in a stage are done, the stage closes (`historian.py:158-173`). User-issued `updateProtocolEvent` commands and AI events both flow through the same path.

This model is rigid — sparse numbering and arbitrary skip-ahead are awkward — which is exactly the pain point that motivated the flat instruction-ID model in [v2](v2.md).

## Entry point and how it runs

The arbiter is launched as a subprocess by [monitor_relay](../monitor-relay.md) with `MonitorType="Arbiter"`. The relay's `MONITOR_CMD_DATA_MAP` in `Lumi-AI-Continuous/monitor_relay/monitor.go:110-117` actually points the `Arbiter` slot at `arbiter_v2.py` — to launch v1 you invoke `python protocol_arbiter/arbiter.py` directly with a config JSON.

On boot (`arbiter.py:102-115`) it sets up Kafka, loads the protocol from S3 (or `test.json` in local mode), spawns the curator and historian threads, and emits `Arbiter Started` on the status topic.

## Topics consumed

- `protocol-arbiter-commands` (`arbiter.py:134`) — user commands keyed by experiment ID: `pauseArbiter`, `resumeArbiter`, `stopArbiter`, `updateLabeyes`, `addListener`, `removeListener`, `updateProtocolEvent`. Documented in `protocol_arbiter/README.md`.
- Per-monitor data topics — added dynamically by `addListener` and forwarded into the curator's subscription set (`curator.py:75-87`). Effectively the `MONITOR_DATA_TOPIC` shape, partitioned per monitor.

## Topics published

- `protocol-arbiter-status` (`arbiter.py:111`) — plaintext lifecycle messages.
- `protocol-arbiter-history` (`arbiter.py:299`) — JSON history rows the web app uses to replay an experiment.
- `monitor-events-<arbiterId>` and `monitor-commands-<monitorId>` (`arbiter.py:614, 1359`) — per-experiment fan-out for spawning, configuring, and tearing down monitors via the relay.

## Curator and Historian roles

`DataCurator` (`curator.py`) is a long-running consumer thread. It reads each monitor message, looks up the originating monitor's `type` (`detector` / `tracker` / other), and slots the payload into `history_cache["labeyes"][labeye_id][bucket][timestamp]`. The `history_cache` dict is the shared bus between monitor data and protocol decisions.

`ExperimentHistorian` (`historian.py`) reads the same cache. `process_manual_inputs` walks `history_cache["protocol_actions"]`, marks the corresponding `protocol["stages"][i]["steps"][j]["actions"][k]` complete, and appends a row to the ledger. Every cycle it also calls `upload_ledger` / `upload_protocol`, gated by the `s3` circuit breaker from [Common.resilience](../common.md).

## Tests

`protocol_arbiter/Testing/` holds smoke scripts: `test_historian.py`, `test_start_monitor.py`, `test_update_protocol_event.py`, `test_send_event.py`. They are imperative scripts driven against a real or mocked Kafka rather than pytest fixtures — closer to manual integration probes than unit tests.

## When to read this vs v2

Read v1 when you need to debug a legacy experiment's ledger format, understand the stage/step/action JSON the web frontend still emits, or trace why something in the `monitor-events-<id>` topic looks the way it does. For everything else, jump to [v2](v2.md) — that's what production runs and what active development targets.
